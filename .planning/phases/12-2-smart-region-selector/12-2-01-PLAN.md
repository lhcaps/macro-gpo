<objective>
Build the drag-to-select overlay module (`src/overlays/region_selector.py`) that lets the operator visually pick a screen region, normalized to [0-1] coordinates, and save it to `combat_regions_v2` via the `set_region()` service.

Purpose: Enables visual region definition without requiring manual coordinate entry. Runs as a daemon thread so it never blocks the HTTP server.

Output: `src/overlays/region_selector.py` — a self-contained overlay module with drag capture, normalization, and service integration.
</objective>

Create `src/overlays/region_selector.py` with a `RegionSelectorOverlay` class that implements:

1. **Constructor `__init__(self, window_title: str, region_name: str = "combat_scan")`**:
   - Store window_title and region_name
   - Do NOT call get_window_rect() here — call it fresh in the thread

2. **Daemon thread entry `run()`**:
   - Call `get_window_rect(self.window_title)` fresh — if None, log error and return (do NOT raise)
   - Position a Tkinter Toplevel over the window rect using `win32gui.SetWindowPos`
   - Set `overrideredirect(True)`, `attributes('-topmost', True)`, transparent background color ` '#000001'` (near-black so it is invisible except where drawn)
   - Configure geometry to cover full window: `f"{win_width}x{win_height}+{left}+{top}"`
   - Set cursor to crosshair via canvas

3. **Canvas drag box**:
   - Create a Canvas with `bg='#000001'` (transparent-ish), `cursor="crosshair"`
   - Bind `<Button-1>` to record start point, `<B1-Motion>` to draw live rectangle, `<ButtonRelease-1>` to finalize
   - Enforce 5x5 pixel minimum: if `abs(x2-x1) < 5 or abs(y2-y1) < 5`, discard and reset (do not commit)
   - Clamp all coordinates within window bounds: `max(left, min(x, right))` etc.

4. **Visual feedback**:
   - Draw live rectangle: orange outline `#FF9500`, line width 2, no fill
   - Draw dimension label: white text on semi-transparent black rect below/beside the box, format: `"{w} x {h}px"` (pixel dimensions of the drag box)
   - Update label on every motion event

5. **Key bindings**:
   - Bind `'<Return>'` to `self._confirm()` — calls `self.result_event.set()` with `{'action': 'confirm', 'x1': ..., 'y1': ..., 'x2': ..., 'y2': ...}`
   - Bind `'<Escape>'` to `self._cancel()` — calls `self.result_event.set()` with `{'action': 'cancel'}`
   - Bind `'<Destroy>'` on root window to also call `self._cancel()` (covers window close button)

6. **Normalization**:
   - On `_confirm()`: compute normalized coords:
     ```python
     norm_x1 = (x1 - left) / win_width
     norm_y1 = (y1 - top) / win_height
     norm_x2 = (x2 - left) / win_width
     norm_y2 = (y2 - top) / win_height
     ```
   - Clamp each to [0.0, 1.0]

7. **Async result delivery via threading.Event**:
   - `self.result_event = threading.Event()`
   - `self.run()` blocks on `self.root.wait_window()` after starting mainloop
   - After `wait_window()` returns, the caller's thread reads `self.result_event.is_set()` and `self.result_data`

8. **Styling choices (agent's discretion)**:
   - Border color: `#FF9500` (orange, matches game UI accent)
   - Line width: 2
   - Label font: `("Segoe UI", 10, "normal")`
   - Label bg: `#1A1A1A` with 80% opacity (approximated via canvas rect)
   - Label fg: `#FFFFFF`

9. **No service import at module level** — import `set_region`, `save_config`, `load_config` inside the `_save_and_close()` method to keep the overlay module self-contained and testable.

The class should have these public methods:
- `run()` — launches Tk mainloop in a daemon thread, returns immediately (caller's thread continues)
- `get_result(timeout=None)` — blocks until result available or timeout, returns `{'action': 'confirm', 'area': [nx1,ny1,nx2,ny2]}` or `{'action': 'cancel'}` or `None` on timeout

The class should NOT call `set_region` or `save_config` directly — it only computes normalized coords and returns them. The caller (backend command handler) is responsible for calling `set_region()` + `save_config()` + `load_config()`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| overlay → game window | Overlay covers game window rect; untrusted pixel coords come from user drag input |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12.2-01 | Spoofing | Malicious window with same title | mitigate | `get_window_rect` already validated by Phase 12.1; overlay uses config's `game_window_title` which is trusted |
| T-12.2-02 | Tampering | Drag coords out of [0-1] range | mitigate | Clamp normalized coords to `[0.0, 1.0]` after normalization; `set_region()` calls `validate_area()` which rejects out-of-range values |
| T-12.2-03 | Repudiation | User claims accidental region save | mitigate | 5x5 minimum threshold discards accidental clicks; Esc cancels without calling `set_region()` |
| T-12.2-04 | Information Disclosure | Region data exposed in logs | accept | Region coords are game UI coordinates, not user data; no PII involved |
| T-12.2-05 | Denial of Service | Overlay thread leaks if window not found | mitigate | If `get_window_rect()` returns None, log error at WARNING level and return early without blocking HTTP server |
</threat_model>
